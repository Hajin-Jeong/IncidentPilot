## 2. 문제 정의

### AS-IS (현재)
- 장애 발생 시 "이거 전에 어떻게 해결했지?" → 슬랙/컨플루언스 검색에 5~15분 소비
- 런북이 있어도 찾기 어렵고, 있는지조차 모르는 경우 다수
- 과거 유사 장애 이력이 개인 기억에 의존
- 장애 대응 중 체크리스트 없이 머릿속으로 진행 → 단계 누락 위험

### TO-BE (목표)
- 장애 정보 입력 → 30초 내 관련 런북 자동 매칭
- 단계별 체크리스트로 빠짐없이 대응
- 과거 유사 장애 이력을 즉시 참조
- MTTR(평균 복구 시간) 단축에 기여

## 3. 핵심 기능 (MVP Scope)

### 기능 1: 장애 입력 → 런북 자동 매칭

**입력 방식 (2가지 지원)**
- **A. 로그파일 업로드 (데모 메인)**: 로그 파일(.log, .txt)을 드래그앤드롭 또는 파일 선택으로 업로드 → AI가 로그 원본에서 핵심 에러를 자동 추출 → 런북 매칭
- **B. 텍스트 직접 입력 (보조)**: 에러 메시지, 알림 내용을 직접 텍스트로 입력 → 런북 매칭

**처리**: AI가 입력된 장애 정보를 분석하여 사전 등록된 런북 목록과 매칭 (유사도 기반)

**출력**
- 관련도 높은 런북 1~3개를 랭킹과 함께 표시
- 매칭 신뢰도(%) 표시, 매칭 근거 요약
- (로그파일 업로드 시) AI가 추출한 핵심 에러 요약 별도 표시

### 기능 2: 체크리스트 기반 대응 가이드
- 매칭된 런북을 단계별 체크리스트로 변환하여 표시
- 각 단계 완료 체크 가능
- 현재 진행 단계 하이라이트
- 각 단계별 예상 소요시간 표시
- 타임라인 자동 기록 (몇 시 몇 분에 어떤 단계 완료)

### 기능 3: 과거 유사 장애 이력 조회
- 입력된 장애 정보와 유사한 과거 장애 목록 표시
- 각 이력에 발생일, 원인, 해결방법, 소요시간 요약
- "이 장애는 3번 발생했고 평균 해결 시간은 23분입니다" 같은 인사이트

## 4. 화면 구성

### 4-1. 메인 대시보드
- 장애 입력 영역: 로그파일 드래그앤드롭 존 + 텍스트 직접 입력 탭 전환
- "분석 시작" 버튼
- 최근 장애 이력 간략 목록

### 4-2. 분석 결과 화면 (3-Column 레이아웃)
- **좌측**: 매칭된 런북 목록 (랭킹, 신뢰도)
- **중앙**: 선택한 런북의 체크리스트 (대응 가이드)
- **우측**: 유사 과거 장애 이력 패널

### 4-3. 장애 타임라인 뷰
- 체크리스트 진행 중 자동 기록되는 타임라인
- 각 단계 시작/완료 시각
- 전체 대응 소요시간

---

## 5. 목데이터 설계

### 런북 목데이터 (5~6개)
1. DB 커넥션 풀 고갈 대응
2. API 서버 5xx 에러 급증 대응
3. Redis 메모리 초과 대응
4. 디스크 용량 부족 대응
5. SSL 인증서 만료 대응
6. Kafka 컨슈머 랙 증가 대응

### 과거 장애 이력 목데이터 (10~15건)
- 각 이력에 포함: 발생일시, 서비스명, 증상, 원인, 해결방법, 소요시간, 심각도(P1~P4)

## 7. 데모 시나리오 (3분)

> **상황**: 새벽 3시, 온콜 엔지니어에게 슬랙 알림이 옵니다.
> "payment-service 5xx 에러 급증, 에러율 15% 돌파"

1. **[0:00~0:30]** 대시보드에 실제 로그파일(.log)을 드래그앤드롭으로 업로드
2. **[0:30~1:00]** AI가 로그에서 핵심 에러 자동 추출 → "API 서버 5xx 에러 급증 대응" 런북 자동 매칭 (신뢰도 92%)
3. **[1:00~1:30]** 체크리스트 단계별로 진행 — 각 단계 체크하며 타임라인 자동 기록
4. **[1:30~2:15]** 우측 패널에서 과거 유사 장애 3건 확인 — "지난번엔 DB 커넥션 문제였다"는 인사이트 발견
5. **[2:15~3:00]** 대응 완료 후 타임라인 리뷰, 확장 가능성(슬랙 연동, 포스트모템 자동 생성) 언급
